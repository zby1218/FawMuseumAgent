# FawMuseumAgent 项目设计文档

## 零、代码风格要求

本项目所有代码须遵守以下风格约定，优先级高于个人偏好。

### 0.1 模块化

- 每个 `.py` 文件只做一件事，文件名即职责描述（`intent_router.py`、`llm_client.py`）
- 禁止跨模块直接访问内部实现，只通过公开函数/类交互
- LangGraph 节点函数独立成文件，不在一个文件里堆多个节点

### 0.2 简洁

- 函数超过 40 行需要重新考虑是否应该拆分
- 不写只用一次的 helper（直接内联）
- 不写"以后可能用到"的参数或配置项
- 不写防御性 `try/except`：捕获 `Exception` 必须有明确的处理逻辑，不能只是 `logger.warning` 后继续

### 0.3 类型标注

- 所有公开函数必须有完整类型标注（参数 + 返回值）
- LangGraph State 字段全部标注，使用 `TypedDict`
- 不用 `Any` 除非对接外部库确实无法标注

### 0.4 注释

- 默认不写注释
- 只在以下情况写注释：隐含约束、反直觉行为、绕过某个外部 bug
- 不写解释"做了什么"的注释（代码本身就是文档）
- 不写 docstring，除非是对外暴露的公共 API

### 0.5 配置与硬编码

- 关键词、阈值、模型名称、prompt 模板全部放 `config/` 或 `.env`，不硬编码在业务代码里
- 业务代码读取配置，不直接 import yaml 文件（通过统一的 `config/loader.py`）

### 0.6 测试

- 单元测试不依赖网络（LLM 调用 mock）
- 评测（eval）与单元测试分开放：`tests/unit/` vs `tests/eval/`
- 每个 Skill 有对应的独立测试，不依赖完整 Agent 启动

## 一、项目要求

### 1.1 项目定位

汽车博物馆讲解机器人 Agent 系统。访客通过语音或文字与机器人交互，机器人理解意图后完成讲解、导航、视频播放、展品识别等任务。

---

### 1.2 功能要求

| 能力 | 说明 |
|------|------|
| 意图识别 | 将自然语言映射为结构化意图，支持一句话多意图 |
| 多轮对话 | 跨轮保持上下文，支持追问和指代消解 |
| 展品讲解 | 结合当前展品信息生成语音友好的讲解内容 |
| Skill 调用 | 播放视频、导航、展示技术参数、高亮部件等 |
| 动作执行 | 点头、转身、挥手等动作指令低延迟直达机器人 SDK |
| 多模态输入 | 语音（ASR）+ 图像 + 视频（VLM）联合处理，统一转为文本后处理 |
| 记忆系统 | 三层记忆：工作记忆（当次对话）、会话记忆（本次参观）、长期记忆（回头客偏好） |

意图类型（完整枚举）：

| 类型 | 示例 | 处理路径 |
|------|------|---------|
| `action` | 点头、转身、过来 | 关键词拦截 → 机器人 SDK |
| `exhibit_qa` | 这辆车是什么年代的？ | LLM → TTS |
| `chitchat` | 你叫什么名字？ | LLM → TTS |
| `skill` | 播放这辆车的视频 | LLM Function Call → Skill |
| `image_identify` | 这是什么车？（附图或视频帧） | VLM → 展品识别 → LLM |
| `gesture_action` | 摄像头检测到指向动作 | VLM → 机器人 SDK |
| `video_understand` | 看这段视频，里面的车是什么型号？ | VLM → 展品识别 → LLM |
| `qr_scan` | 扫码请求 | QR 解析 → Skill |

---

### 1.3 技术要求

| 模块 | 选型 | 约束 |
|------|------|------|
| Agent 编排 | LangGraph v1.x | 锁定大版本，不随意升级 |
| LLM / VLM | `qwen3-vl-flash-2026-01-22`（云端 API） | 不依赖本地 vLLM；视频/图像/文本统一模型 |
| 意图输出 | Pydantic structured output | 不手动解析 JSON |
| 记忆持久化 | LangGraph MemorySaver + Mem0 | 会话级 + 长期级分离 |
| 包管理 | uv | pyproject.toml + uv.lock 锁版本 |
| 异步执行 | Python asyncio | 无依赖意图并行，有依赖意图串行 |
| 可观测性 | LangSmith 或 Phoenix | 每个节点输入输出可追踪 |

延迟目标：

| 路径 | 目标 |
|------|------|
| 动作指令（关键词命中） | < 50ms |
| LLM 意图分类 | < 500ms |
| LLM 讲解首 token（流式） | < 1s |

### 1.4 设计原则

**简洁优先（Simplicity First）**
- 最少代码解决问题，不写投机性功能
- 单次使用的逻辑不做抽象
- 如果 100 行能写完，不写 300 行

**模块化（Modular）**
- 每个模块职责单一，边界清晰
- 模块间通过明确的接口交互，不互相依赖内部实现
- 意图识别、Skill、记忆、编排各自独立，可单独测试

**可评测（Evaluable）**
- 意图识别模块有独立测试集，评测覆盖率和准确率
- 关键路径有 pytest 覆盖，测试不依赖网络（mock LLM）
- Skill 可以在不启动完整 Agent 的情况下独立调用验证

**可迁移（Portable）**
- 不绑定特定机器人硬件 SDK，动作执行通过抽象接口调用
- LLM 供应商可替换（Anthropic → OpenAI 只改配置，不改业务代码）
- 配置驱动（`config/intents.yaml`），关键词和意图类型无需改代码

**不臃肿（Lean）**
- 不引入未用到的依赖
- 不做未被要求的功能（Skill 沉淀、手势识别等留到后期）
- 每次 PR 只做一件事，改动可追踪

---

### 1.5 评测要求

意图识别模块需满足：

- 提供 `tests/eval/intent_cases.yaml`，包含至少 30 条带标注的输入样本
- 可通过 `pytest tests/eval/` 运行，输出准确率报告
- 关键词拦截层准确率 100%（词表有限，可穷举验证）
- LLM 层意图分类准确率 ≥ 90%（对标注集）
- 多意图用例单独一组，验证 `depends_on` 字段正确性

---

### 1.6 迁移要求

- **硬件迁移**：`action_executor` 接收统一的动作名称字符串，硬件 SDK 调用封装在 `skills/robot_sdk.py` 一处，换硬件只改这一个文件
- **LLM 迁移**：LLM/VLM 调用封装在 `graph/nodes/llm_client.py`，切换供应商只改 `LLM_MODEL` 和 `API_KEY` 环境变量
- **知识库迁移**：展品知识以结构化 JSON/YAML 存储在 `config/exhibits/`，不硬编码在 prompt 里
- **部署迁移**：无状态节点（`intent_router`、`llm_agent`）可独立部署为服务；有状态部分（MemorySaver）可替换为 Redis/PostgreSQL backend
